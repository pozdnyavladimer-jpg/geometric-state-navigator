# PROMOTION RULES

This document defines how ideas move through the GitCube system.

Canonical path:

gitcube-lab
→ geometric-state-navigator
→ gitcube-os

This is the only valid promotion direction.

---

## 1. Core Principle

Not every idea deserves to become runtime.

The system promotes only what is:

- testable
- interpretable
- stable enough
- aligned with canonical state language

Lab may invent freely.
Navigator formalizes meaning.
OS implements only accepted logic.

---

## 2. Repository Roles

### gitcube-lab

Purpose:
- experiments
- unstable simulations
- speculative filters
- early pattern imports
- adversarial stress tests

Allowed:
- temporary naming
- unstable thresholds
- incomplete logic
- exploratory code

Not allowed:
- to define final system law

---

### geometric-state-navigator

Purpose:
- define canonical meaning
- define state semantics
- define threshold philosophy
- define bindu logic
- translate experiments into stable system language

Allowed:
- canonical documentation
- canonical metric definitions
- mapping documents
- validated mathematical interpretation

Not allowed:
- to become a dumping ground of raw experiments

---

### gitcube-os

Purpose:
- execute stable logic
- run loops
- apply bindu decisions
- store memory
- compress memory
- operate in environments

Allowed:
- minimal robust implementations
- clean runtime code
- stable loop behavior

Not allowed:
- to invent new theory directly

---

## 3. Promotion Stages

Every idea should exist in one of the following stages.

### Stage 0 — Experimental

Location:
- gitcube-lab

Meaning:
- raw idea
- not yet canonical
- may fail
- may be renamed
- may be removed

---

### Stage 1 — Interpreted

Location:
- geometric-state-navigator

Meaning:
- the idea has been translated into system language
- it has canonical meaning
- it is linked to state / metrics / thresholds

At this stage, the question is no longer:
"does this look interesting?"

The question becomes:
"what does this mean inside the GitCube state system?"

---

### Stage 2 — Operational

Location:
- gitcube-os

Meaning:
- the idea is stable enough to run
- it has executable form
- it affects runtime behavior

At this stage, simplicity is preferred over richness.

---

## 4. Promotion Gate Questions

Before an idea is promoted from Lab to Navigator,
it must answer all of the following:

1. What problem does it solve?
2. Which canonical axis does it affect?
3. Which canonical metric does it affect?
4. Is it a primary concept or a derived signal?
5. Does it improve stability, coherence, or survivability?

If these answers are unclear, the idea stays in Lab.

---

## 5. Navigator Promotion Questions

Before an idea is promoted from Navigator to OS,
it must answer:

1. Can it be implemented simply?
2. Does it reduce runtime ambiguity?
3. Does it improve decision quality?
4. Can it fail safely?
5. Is it stable enough to become part of the loop?

If the answer is no, it remains Navigator-only.

---

## 6. What Can Be Promoted

These can be promoted:

- metric definitions
- threshold rules
- bindu logic
- reroute logic
- memory policy
- environment rules
- stable pattern mapping

These should only be promoted if clearly defined.

---

## 7. What Should Usually Stay in Lab

These usually remain experimental unless strongly validated:

- metaphoric layers without executable effect
- speculative resonance terms
- raw notebook variables
- one-off heuristics
- visually appealing but behaviorally weak logic
- unstable filters that do not survive stress

Lab is allowed to be rich.
OS must remain clean.

---

## 8. Canonical Translation Rule

A Lab concept may only move upward if it can be rewritten in canonical language.

That means it must be expressible through at least one of:

- pressure
- flow
- structure
- balance
- law
- future

or through canonical metrics:

- shadow
- coherence
- target_fit
- vitality

If it cannot be translated into this language,
it is not ready for promotion.

---

## 9. Minimal Promotion Checklist

A promotion candidate should include:

- short description
- source repo and file
- canonical interpretation
- affected metrics
- affected thresholds
- expected runtime effect
- known failure mode

Without this, promotion is incomplete.

---

## 10. Example Promotion

Example:
A Lab notebook introduces `coherence_frequency`.

Correct promotion path:

Stage 0:
- appears in gitcube-lab as experimental signal

Stage 1:
- in Navigator, it is defined as a derived coherence-related signal
- it is documented and linked to canonical coherence logic

Stage 2:
- only if useful, OS uses it in a simplified way
- for example as an optional secondary gate

This is valid promotion.

Incorrect promotion:
- copying raw notebook logic directly into runtime

---

## 11. Runtime Protection Rule

OS must reject promotion if:

- meaning is unclear
- thresholds are unstable
- implementation is too complex
- failure behavior is unknown
- the idea creates more ambiguity than clarity

Runtime is not a research notebook.

---

## 12. Memory Promotion Rule

Only transitions that survive runtime should become memory.

This applies not only to episode memory,
but also to architectural memory at the repository level.

Meaning:
- Lab generates candidates
- Navigator interprets candidates
- OS keeps only what survives execution

This is the architectural equivalent of memory compression.

---

## 13. Promotion Status Vocabulary

Use these labels consistently:

- experimental
- interpreted
- canonical
- runtime-ready
- promoted
- rejected
- deprecated

This helps both humans and AI understand repo maturity.

---

## 14. Minimal Law

Nothing enters OS unless it first becomes clear in Navigator.

---

## 15. One-line Summary

Lab invents.
Navigator explains.
OS keeps only what survives.
